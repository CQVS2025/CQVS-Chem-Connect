1. What it does, in one sentence

An admin can generate a shareable URL for a specific product. Anyone with that URL can submit a review without authentication, but the review is clearly marked as “Reviewer” rather than “Verified buyer” on the product page, and does not count toward the headline star rating or the Google rich-result aggregate. Public reviews still display on the page below the verified ones, adding texture without moving the headline number.

2. Why this is Phase 2 work

Phase 1 ships a verified-buyer-only review flow: every approved review is tied to a real order_id via a magic-link token. That keeps the trust signal clean and the AggregateRating defensible. Public share links break the verified-buyer guarantee for a subset of reviews - which means they need a separate badge, separate aggregate math, and separate spam protection.

None of this is hard, but layering it onto the verified-only model from day one would have made Phase 1 unnecessarily heavier. So Phase 1 ships first; Phase 2 adds the public-link path on top of the existing schema with a single migration and a small admin surface.

Why the design is locked now: doing the four design calls below at build time would risk a small refactor (especially the verified-only-aggregate decision, which lives at the SQL layer). Locking now means the Phase 2 build is mostly a configuration job over the Phase 1 schema.

3. Locked decisions (round 1 + round 2)

Four decisions, all closed. Three approved on the first pass; #4 was redirected by Jonny in round 2.

#DecisionLocked outcome
1 Verified vs. unverified badge on the card Approved Magic-link and manual-entry reviews keep the “Verified buyer” badge with the green tick. Public-link reviews show “Reviewer” in muted grey. The two are visually distinct so a careful reader can tell the difference at a glance.
2 Default share-link settings Approved New links default to 30 days expiry / 50 uses. Both adjustable per-link at generation time. No-expiry option available for things like the email-signature link Jonny mentioned. Max-uses can also be set to unlimited per-link.
3 Spam protection Approved 3 submissions per IP per hour rate limit on the public submit endpoint, plus an invisible honeypot field for bots. IP is sha256-hashed before storage - never raw.
4 Star rating math Redirected by Jonny Verified-only. The headline average shown on the product page AND the AggregateRating JSON-LD count only verified-buyer reviews. The ≥3-review threshold for emitting JSON-LD is also verified-only. Public-link reviews appear on the page in the cards but never affect the headline number or the rich result.
Why Jonny redirected #4: the badge protects integrity for someone reading the product page carefully, but Google scrapes the aggregate number and that’s what most prospects see first. A competitor with a share link could move the headline rating, and a sabotaged 3.8 average is worse than no stars at all for high-trust B2B buyers. Slower SEO ramp on new products is the right trade against that exposure.

4. The four real-world situations this handles

SituationWhat happens
Genuine repeat customer who didn’t get the auto-email Maybe they ordered before reviews shipped, or the email landed in spam. They click the share link, type their review, submit. Lands in the moderation queue. Once approved, displays as “Reviewer” on the product page - adds page texture but doesn’t move the headline number.
Verified buyer forwards link to a colleague Plant manager forwards the URL to a site supervisor. Both have used the product, both want to weigh in. Both submissions land via the same share link, both go through moderation, both display as “Reviewer”. Magic-link tokens (one-use, tied to one order) couldn’t support this without complex re-issuance logic; share links handle it naturally.
Real user who isn’t on the order list Procurement bought the product but the actual user is a different person. They have a real opinion but no order_id to verify against. Submits via the share link, displays as “Reviewer” - honest representation: they used the product but Chem Connect can’t prove they bought it.
Drive-by spam / competitor sabotage Worst case. The per-IP rate limit and honeypot catch automated traffic; the admin moderation queue catches manual abuse. Even if one slips through, it can’t move the headline rating or the rich result because public reviews are excluded from the aggregate math.

5. Schema additions

A single Phase 2 migration. Builds on the existing 046_customer_reviews.sql schema; no breaking changes.

-- 048_review_share_links.sql

-- 5.1 Extend the existing source enum
alter type review_source add value if not exists 'public_link';

-- 5.2 Track whether a review counts as a verified-buyer review.
-- Magic-link / manual entries set this true; public-link entries set it false.
-- Used by the aggregate-math layer to gate the headline rating + JSON-LD.
alter table reviews
  add column if not exists verified_buyer boolean not null default false;

-- 5.3 Backfill existing rows. Phase 1 has only magic_link + manual sources,
-- both of which are verified.
update reviews
   set verified_buyer = true
 where source in ('magic_link', 'manual')
   and verified_buyer = false;

-- 5.4 Public share-link records
create table review_share_links (
  id uuid primary key default uuid_generate_v4(),
  slug text not null unique,                -- short URL-safe string ('xj8w-2k4f')
  product_id uuid not null references products on delete cascade,
  created_by uuid references auth.users,
  expires_at timestamptz,                   -- nullable: null = no expiry
  max_uses integer,                         -- nullable: null = unlimited
  used_count integer not null default 0,
  is_active boolean not null default true,
  created_at timestamptz not null default now()
);

create index review_share_links_slug_idx
  on review_share_links (slug)
  where is_active = true;

create index review_share_links_product_id_idx
  on review_share_links (product_id);

-- 5.5 Per-IP rate-limit tracking. IP hashed before storage.
create table review_share_submission_attempts (
  id uuid primary key default uuid_generate_v4(),
  ip_hash text not null,
  share_link_id uuid references review_share_links on delete set null,
  created_at timestamptz not null default now()
);

create index review_share_submission_attempts_ip_idx
  on review_share_submission_attempts (ip_hash, created_at desc);

-- 5.6 RLS
alter table review_share_links enable row level security;
alter table review_share_submission_attempts enable row level security;

create policy "Admins can manage share links"
  on review_share_links for all
  using ((auth.jwt() -> 'user_metadata' ->> 'role') = 'admin')
  with check ((auth.jwt() -> 'user_metadata' ->> 'role') = 'admin');

-- Submission-attempt rows are written + read only by the service role
-- (the public submit endpoint). No anon / authenticated access.

6. Aggregate math – the key change

The single piece of logic that needs careful handling. Three distinct concepts that look similar but mean different things.

WhatSource filterThresholdUsed for
Headline average (sticky chip near the buy button + reviews-block summary) verified_buyer = true AND status = 'approved' None - show whatever count exists The “4.6 · 23 reviews” pill that buyers see first.
JSON-LD AggregateRating + Review[] verified_buyer = true AND status = 'approved' ≥ 3 verified reviews Google rich-result eligibility - the star result in search.
Visible review cards (marquee / inline grid) All status = 'approved' (verified + public mixed together) None The reviews section content. Public-link reviews show with the “Reviewer” badge.

Implementation

  • Existing getApprovedReviewAggregate(productId) in lib/reviews/queries.ts → add .eq('verified_buyer', true) filter.
  • New getReviewsForDisplay(productId, limit) in the same file → returns ALL approved reviews regardless of source, ordered by published_at desc. Used by the product page to populate the cards.
  • Existing getApprovedReviewsForProduct(productId, limit) stays as-is (verified-only) - that one feeds the JSON-LD Review[] array.
Net effect on the card list: the page shows verified and public reviews intermixed by date. The verified ones have the green tick; the public ones have the muted grey badge. The headline number above the list is verified-only, so a buyer skimming the page sees an honest aggregate but still gets the full content.

7. Admin UI

7.1 New “Generate share link” flow on /admin/marketing/reviews

  • New button next to “Add review manually”: “Generate share link”.
  • Opens a modal with:
    • Product (dropdown - required)
    • Expiry (radio: 7 days / 30 days / 90 days / Never - default 30 days)
    • Max uses (input - default 50, blank = unlimited)
    • Generate button
  • After generation, modal swaps to a result panel:
    • The generated URL: https://www.cqvs-chemconnect.com.au/reviews/share/xj8w-2k4f
    • Copy-to-clipboard button
    • “Open in new tab” link to preview the public page

7.2 New “Share links” tab on the same page

Adds a fourth tab next to Pending / Approved & Live / Rejected. Lists all share links the admin has generated.

  • Columns: slug (last 6 chars), product, created date, expiry, used / max, status (active / expired / revoked), actions.
  • Per-row actions:
    • Copy URL - one-click copy of the shareable link
    • Revoke - sets is_active = false; the link stops working immediately
    • View submissions - filters the moderation queue to reviews submitted via that link
  • Empty state: short copy explaining what share links are and a button to generate the first one.

8. Public submission page

8.1 URL

/reviews/share/[slug]

8.2 Server-side checks on page load

  1. Slug exists in review_share_links
  2. is_active = true
  3. expires_at is null or in the future
  4. max_uses is null or used_count < max_uses

If any check fails, render a friendly “This link is no longer active” message with a link back to the product page. Same visual style as the existing magic-link expired/consumed states.

8.3 The form – how it differs from the magic-link form

Most of the existing form is reused. The differences are:

  • Product is fixed (from the slug). No dropdown.
  • Two extra required fields: email and company / role. These give moderation context that magic-link reviews get for free (via the order’s buyer record).
  • No “Verified buyer” copy anywhere in the form.
  • Different consent line: “I confirm this is my honest opinion based on my use of the product. I understand it’ll be displayed as a public reviewer, not a verified buyer, and won’t count toward the product’s star rating average.”
  • Honeypot field: an invisible <input name="website"> styled display:none. Bots auto-fill, humans never see it. Submissions where this field has any value are silently rejected.

8.4 On submit (POST /api/reviews/share/submit)

  1. Per-IP rate limit check: sha256-hash the IP from x-forwarded-for, query review_share_submission_attempts for the last hour. If ≥ 3 → return HTTP 429.
  2. Honeypot check: if the website field has any value, return HTTP 200 with a fake-success body but don’t insert anything. Don’t signal failure to bots - they shouldn’t learn the trap is there.
  3. Re-validate the share link is active, not expired, not over max_uses.
  4. Insert into reviews with source = 'public_link', verified_buyer = false, status = 'pending'.
  5. Increment used_count on the share-link row.
  6. Insert review_share_submission_attempts row for the IP.
  7. Insert review_audit_log row (action 'submitted', source noted in detail).

9. Product page display

When a product has BOTH verified and public reviews, here’s how the product page renders:

Product page: Sticky stars + average chip near buy button "4.7 · 12 reviews" <-- verified-only count + average Reviews section below: Header chip: "4.7 · 12 reviews" <-- same verified-only number Review cards (intermixed by date): - Marcus T., Brisbane [Verified buyer] <-- contributed to 4.7 - Plant Manager, NSW [Verified buyer] <-- contributed to 4.7 - Reviewer at NSW site [Reviewer] <-- visible only - Anonymous from Perth [Verified buyer] <-- contributed to 4.7 ... JSON-LD: AggregateRating ratingValue=4.7 reviewCount=12 Review[] - up to 10 most recent VERIFIED reviews

9.1 Badge logic

  • Verified - emerald pill with a ShieldCheck icon. Same look as today.
  • Reviewer - muted grey pill, no icon. Tooltip on hover: “This reviewer used the product but isn’t tied to a Chem Connect order.”

9.2 Edge cases

  • Product has 0 verified, 5 public reviews: headline chip shows nothing (no verified average to display); cards still render with all 5 public reviews and the empty-state message above the cards (“No verified buyer reviews yet”). JSON-LD AggregateRating is not emitted.
  • Product has 2 verified, 5 public reviews: headline shows “X.Y · 2 reviews”. JSON-LD AggregateRating still NOT emitted (below the ≥3 verified threshold). Cards show all 7.
  • Product has 4 verified, 0 public: identical to today’s Phase 1 behaviour. Nothing to render differently.

10. Spam guards (locked)

GuardSpec
Per-IP rate limit 3 submissions per hour per IP, sha256-hashed before storage. Bypass returns HTTP 429.
Honeypot field <input name="website"> styled display:none. Filled = silent reject (HTTP 200 with fake-success body).
Email format validation Standard email regex. Plus a small disposable-email blocklist (10minutemail, mailinator, guerrillamail, tempmail, etc.). Maintained in lib/reviews/disposable-emails.ts; easy to extend.
Per-link cap max_uses ceiling on each share link. Default 50 unless admin overrides. Once hit, the link returns “no longer active”.
Admin moderation Every public-link submission lands as status = 'pending'. Admin reviews and approves / rejects via the same queue used for magic-link reviews. Source is shown on the moderation card so admin can apply higher scrutiny to public submissions.
Verified-only aggregate (decision #4) Even if a public review slips moderation, it cannot move the headline rating or the Google rich-result. This is the strongest defence and the reason Jonny pushed back on the original recommendation.

12. What this leaves out (deliberate)

These were considered and explicitly excluded from the Phase 2 scope:

Excluded itemWhy
Out Email confirmation step before public-link reviews post Adds friction for the reviewer without enough payoff given that admin moderation already catches everything before publication. If spam volume turns out to be high in practice, this can be added in Phase 3 as a per-link toggle.
Out CAPTCHA on the public submit form Friction-cost is high for legitimate reviewers; honeypot + IP rate limit + admin moderation is sufficient at this scale. Revisit if real-world spam patterns warrant it.
Out Per-product max active share links Admin can revoke any link manually; no need to enforce a count. If admins start piling up dozens of overlapping links the moderation queue itself will surface the issue.
Out Bulk-revoke / link analytics dashboard Phase 3+ if usage warrants. The Phase 2 list view shows used / max counts which is enough to spot the active links at a glance.
Out Public-link reviews counting toward the AggregateRating Locked decision #4. Verified-only aggregate is the explicit trade for sabotage protection.
