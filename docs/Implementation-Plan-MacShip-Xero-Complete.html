Add MacShip columns to existing tables and create lead time tables.

  • New table: lead_time_product_warehouse — most specific override

    • warehouse_id UUID FK
    • manufacturing_days integer
    • buffer_days integer

    • notes text
    • UNIQUE(product_id, warehouse_id)
  • New table: macship_failed_updates — for flagging failed consignment sync

B.2 MacShip API Client

New file: lib/macship/client.ts

Typed API client for all MacShip endpoints. Reads env vars at runtime to switch test/production.

  • getRoutes(payload) — POST /apiv2/routes/returnRoutes
    Sends: pickup suburb/postcode/state, delivery postcode, items (packaging size, weight, dimensions, DG class), tailgate YES/NO. Returns array of carrier options with prices. We always select minimum price.
  • createConsignment(payload) — POST /apiv2/consignments/createConsignment
    Sends: carrier ID, pickup/delivery details, items, scheduled pickup date (from lead time calculation). Returns: consignment ID, tracking info.
  • updateConsignment(id, payload) — PUT /apiv2/consignments/{id}
    Used to update pickup date when lead times change.
  • manifestConsignment(consignmentId) — POST /apiv2/manifests/manifest
    Called when admin dispatches order. Notifies carrier to pick up.
  • getConsignment(id) — GET /apiv2/consignments/getConsignment?id={id}
    Polls for tracking updates and POD.
  • getLabels(consignmentId) — returns printable label PDF URL
Base URL: https://api.macship.com.au (production) | https://staging-api.macship.com.au (test)

Include: request/response TypeScript types, error handling with meaningful messages, automatic test/prod switching from MACSHIP_MODE=test|production env var.

B.3 Lead Time Calculation Logic

New file: lib/macship/lead-time.ts

Pure utility functions — no side effects, easy to test.

  • getLeadTime(productId, warehouseId) — resolves the lead time using inheritance:
    1. Check lead_time_product_warehouse for exact product+warehouse match
    2. Fall back to lead_time_warehouse for warehouse default
    3. Fall back to lead_time_global
    4. Hardcoded fallback: 5 business days if nothing configured
  • calculatePickupDate(manufacturingDays, bufferDays, useBusinessDays, warehouseState)
    • Start from today (order placement date)
    • Add manufacturing + buffer days
    • Skip weekends if useBusinessDays = true
    • Skip AU public holidays for the warehouse's state (use a maintained holiday list: VIC, QLD, NSW, SA, WA)
    • Roll forward if result lands on weekend or public holiday
    • Return: ISO date string
  • getOrderPickupDate(orderItems, warehouseId)
    • For each item: resolve lead time for that product + warehouse
    • Return the longest lead time result (governing product)
    • Also return which product_id is governing (for display in admin)

Edge Cases

ScenarioHandling
Pickup date on weekend/holidayRoll forward to next business day using warehouse state holidays
No lead time configured anywhere5-day fallback; flag order in admin with warning badge
Multiple products, different lead timesUse maximum; note governing product on order record
Calculated date is in the pastUse today as minimum; flag for manual review

B.4 Warehouse Selection Logic

New file: lib/macship/warehouse-selector.ts

  • selectWarehouse(cartItems, deliveryPostcode)
    1. For each cart item: query product_warehouses to find warehouses that carry it
    2. Find warehouses that carry ALL items in the cart
    3. If multiple qualify: score by proximity to delivery postcode (postcode distance lookup table for AU states — exact postcode proximity requires MacShip route call, so use state-level scoring as approximation: same state = score 1, adjacent state = score 2, etc.)
    4. Return single winning warehouse
    5. If no single warehouse carries all products: return warehouse with most items + flag for admin split-order review
  • Cache postcode → state mapping using a lightweight AU postcode range lookup (no external API needed)

B.5 Checkout — Live MacShip Quote

New API route: app/api/macship/quote/route.ts

Called client-side when customer enters delivery postcode at checkout (debounced, 500ms).

  • Input: delivery postcode, cart items (product IDs, packaging sizes, quantities), forklift_available
  • Server-side: run warehouse selection logic → get winning warehouse → call MacShip getRoutes → return cheapest carrier + price
  • Output: { shipping_amount, carrier_name, warehouse_id, warehouse_name, estimated_transit_days, pickup_date }
  • If postcode not serviceable: return { serviceable: false } → checkout shows "We can't calculate shipping for your area" + submit button that emails admin

Checkout UI changes (app/(marketplace)/checkout/page.tsx)

  • After postcode field: show shipping quote status (loading spinner → carrier name + AUD amount)
  • Shipping line in order summary: live from quote (replacing current static value)
  • Show estimated delivery date (pickup date + transit days from MacShip response)
  • Unserviceable postcode: overlay message with "Notify Admin" submit button
  • Block checkout submission until valid quote received (or unserviceable path chosen)

B.6 Order Placement — Create MacShip Consignment

Update: app/api/orders/route.ts

After the order is created in Supabase, immediately call MacShip createConsignment.

  • Build MacShip consignment payload from order data:
    • Pickup: winning warehouse address, contact details
    • Delivery: customer's delivery address from checkout
    • Items: packaging size dimensions/weight from packaging_sizes table, DG class from product
    • Tailgate: !forklift_available
    • Scheduled pickup date: from lead time calculation
    • Carrier: ID from the quote response (stored during checkout)
  • On success: save macship_consignment_id, macship_carrier_id, macship_tracking_url, macship_pickup_date to order record
  • On failure: log error, continue order creation (do NOT block order — flag in admin for manual MacShip booking)
  • Also: save macship_governing_product_id (which product has the longest lead time)

B.9 Automatic Lead Time Change → MacShip Sync

New API route: app/api/macship/sync-lead-times/route.ts

Triggered automatically whenever a lead time value is saved in admin. This is the critical feature from Jonny's document.

1
Admin saves a lead time change (global, warehouse, or product-specific). The save action calls the sync endpoint.
2
Find affected orders: Query all orders where macship_dispatched_at IS NULL (not yet dispatched) and the warehouse_id matches the changed lead time scope.
3
Recalculate pickup date for each affected order using the new lead time. Skip if calculated date is in the past — flag those for manual review.
4
Call MacShip updateConsignment for each affected order with the new pickup date. Record success or failure.
5
Return summary to admin UI: "X orders updated. Y orders need manual review." Failed updates are written to macship_failed_updates table and shown as warning badges in the orders list.

Edge case handling

ScenarioHandling
New pickup date is in the pastSkip MacShip update. Flag order for manual review. Show in admin summary.
Order already dispatchedExclude from query entirely. No action.
MacShip update call failsWrite to macship_failed_updates. Show warning badge on order. Do NOT silently fail.
No MacShip consignment on orderSkip (consignment creation may have failed originally). No update possible.

B.10 Tracking Sync

New API route: app/api/macship/sync-tracking/route.ts

Polling endpoint called periodically (or via cron) to sync tracking updates for active shipments.

  • Query orders where macship_dispatched_at IS NOT NULL and status != 'delivered'
  • For each: call MacShip getConsignment → check for status updates / POD
  • If delivered: update order status to 'delivered', save tracking update
  • Display tracking status and POD link in both admin order view and customer dashboard order view
