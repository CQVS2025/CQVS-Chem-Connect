Check .env.local

VariableRequired ValueNotes

3.5 Server Log Verification

While the quote is fetching, watch the dev server terminal. You should see the full request and response logged:

[Machship quote] request: { "companyId": 55514, "fromLocation": {...}, "toLocation": {...}, "items": [{ "itemType": 1 or 3, "name": "Eco Wash", ... }] } [Machship quote] response: { "id": "<uuid>", "routesCount": 1, "firstRoute": { "carrier": { "displayName": "Aramex (ARAMEX)" or "Hi Trans (HITRANS)" }, ... } }

Phase 5. Lead Time Sync After Config Change

This tests the "auto MacShip update" feature requested by Jonny — when a lead time changes, all pending orders should have their pickup dates recalculated and MacShip notified.

1
Go to /admin/warehouses → Lead Times tab → Global Default section
2
Change global default from 3 days to 6 days and click Save
3
Observe the sync result toast that appears after saving
Toast: "X orders updated. Y orders need manual review." (X = orders whose pickup date was recalculated and MacShip notified. Y = orders where the new date fell in the past.)
4
Go to /admin/orders → find your test order → expand it
Pickup date should be updated to reflect the new 6-day lead time. Note: the product with the 2-day override will not change since it has its own rule.
When a real Machship consignment exists, the sync calls POST /apiv2/consignments/updateDespatchDatesForManifest. If that call fails, the order is added to the macship_failed_updates table and shown in the Failed Updates panel on /admin/macship. Order creation and the rest of the flow are never blocked by sync failures.

Phase 8. Edge Cases

ScenarioHow to TestExpected Behaviour
No products mapped to any warehouse Remove all mappings from Product Mapping tab, then add a product to cart and go to checkout Quote returns unserviceable warning. "Continue to Payment" is disabled.
No lead time configured anywhere Delete warehouse lead time. Delete global default row. Place an order. 5-day hardcoded fallback is used. Order shows ⚠ "Used 5-day lead time fallback" badge in admin.
Pickup date lands on a weekend Set lead time so today + days = next Saturday Pickup date auto-advances to the following Monday.
Pickup date lands on an AU public holiday Set lead time to land on a known public holiday (e.g. ANZAC Day 25 Apr) Pickup date rolls forward to next business day after the holiday.
DG product in cart Use a product with classification = "DG" (Dangerous Goods) MacShip quote requests a DG-capable carrier. Carrier options may differ from non-DG orders.
Multiple products, different lead times Add multiple products to cart (one with a 2-day override, one with the 4-day warehouse default) Governing (longest) lead time is used. Pickup date based on 4-day product. Admin shows which product is governing.
MacShip consignment fails Temporarily corrupt the test token in .env.local, place an order, then restore it Order is created successfully. "⚠ Consignment creation failed — book manually" shown in admin orders. Entry in Failed Updates panel on /admin/macship.
Lead time change puts pickup date in the past Set a very short lead time (0 days) on the global default, then save Sync runs. Affected orders reported in "needs manual review" count in the toast. Not silently ignored.

Phase 10. Switch to Production

Only do this after all test phases pass successfully. Switching to production will create real consignments with real carriers and real charges.

Machship — switch from test to production

1
Set MACSHIP_MODE=production in .env.local and your hosting environment (Vercel / Railway etc.)
2
Confirm MACSHIP_PRODUCTION_API_TOKEN and MACSHIP_COMPANY_ID are set
3
Restart the server (or redeploy)
4
Go to /admin/macship — mode badge should show production
5
Place one real test order with a real AU address
6
Confirm the consignment appears in the Machship production dashboard at https://app.machship.com

Xero — switch from Demo Company (Global) to real AU production org

1
Remove or comment out these env vars from .env.local and hosting:
XERO_SALES_TAX_TYPE=NONE
XERO_PURCHASE_TAX_TYPE=NONE
XERO_CURRENCY=AUTO
2
The defaults will kick back in: OUTPUT2 for sales GST, INPUT2 for purchase GST, AUD for currency
3
Disconnect the Demo Company on /admin/xero and reconnect, choosing the real AU org during the OAuth consent screen
4
Update each warehouse's xero_contact_id to point at the real supplier contact in the AU org
5
Place a test order — the invoice will now be created in AUD with proper 10% GST automatically calculated by Xero, and totals will match the platform total exactly
Consignments visible in real Machship + Xero accounts. Carriers notified on dispatch. Real labels generated. AU GST applied automatically.
