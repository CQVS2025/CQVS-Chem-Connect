2. System Architecture

+----------------------------------------------------------+ | BROWSER -- ChemConnect Admin (Next.js 16 / React 19) | | | | /admin/marketing/contacts /marketing/campaigns | | /admin/marketing/sequences /marketing/inbox | | /admin/marketing/dashboard /marketing/settings | +----------------------------------------------------------+ | ^ | Server Actions / fetch | Realtime UI updates v | (SWR / React Query) +----------------------------------------------------------+ | NEXT.JS APP ROUTER -- API routes | | app/api/marketing/* (CC-internal admin API) | | app/api/webhooks/ghl/* (GHL -> CC webhook ingest) | +----------------------------------------------------------+ | ^ | Uses lib/ghl/client.ts | Writes to supabase v | tables (RLS enforced) +---------------------+ +-------------------------+ | GHL API CLIENT | | SUPABASE DB (Postgres) | | lib/ghl/ | | marketing_* tables | | - auth + headers | | + audit_log + events | | - contacts, camp., | +-------------------------+ | - conversations, | | - workflows | +---------------------+ | | HTTPS + Bearer (pit-...) v +----------------------------------------------------------+ | GoHighLevel | | Contacts | Email Engine | SMS/Twilio | Workflows | | (source of truth for marketing data) | +----------------------------------------------------------+

Design principles

  • GHL is source of truth for marketing contacts, sends, sequences, analytics. Local Supabase holds a cache + augmentation layer only.
  • Supabase tables hold durable marketing state (campaign records, event stream, SMS threads, audit log). They survive if GHL webhooks lag or a webhook is missed — we can reconcile.
  • All writes go through a single GHL API client (lib/ghl/client.ts) — centralised auth, retries, error handling. No direct fetch() to GHL scattered around the codebase.
  • Webhooks are idempotent — GHL may retry on 5xx, so every handler keys events by a GHL event ID and upserts.
  • No background workers required — GHL owns scheduling. CC only runs request-response and webhook-response flows, which fit Vercel serverless perfectly.

4. GHL API Client Layer

Folder: lib/ghl/

lib/ghl/ ├── client.ts - Base fetch wrapper: auth header, version header, retry, error types ├── config.ts - Env + constants (base URL, version, rate-limit tuning) ├── errors.ts - GHLApiError, GHLRateLimitError, GHLAuthError classes ├── types.ts - TypeScript types for GHL entities (Contact, Campaign, Workflow, Message, Event) ├── contacts.ts - list, get, create, update, delete, addTags, removeTags, search, bulkUpsert ├── campaigns.ts - createBulkAction, getBulkAction, cancelBulkAction ├── conversations.ts - listConversations, getMessages, sendSms, sendEmail ├── workflows.ts - listWorkflows, getWorkflow, enrollContact, unenrollContact ├── locations.ts - getLocation (used for health checks) └── webhooks.ts - verifySignature, parsePayload (shared by all webhook routes)

Client behaviour

  • Auth: every request gets Authorization: Bearer ${token} and Version: 2021-07-28.
  • Retries: exponential backoff on 429 and 5xx; up to 3 retries; surfaces a typed error after that.
  • Rate-limit guard: reads X-RateLimit-Remaining header; pauses briefly if remaining < 5.
  • Logging: every call logs method, endpoint, status, duration_ms to marketing_audit_log at debug level (opt-in via env flag).
  • Type safety: Zod schemas validate GHL responses at the boundary — protects the rest of the codebase from GHL API changes.

5. Internal API Surface (ChemConnect Routes)

Every admin action goes through a CC-internal route. The route enforces RBAC, then calls the GHL client. This keeps tokens server-side and lets the React admin stay dumb about GHL.

app/api/marketing/ ├── contacts/ │ ├── route.ts GET (list), POST (create) │ ├── [id]/route.ts GET, PATCH, DELETE │ ├── import/route.ts POST (multipart CSV upload + column map) │ ├── tags/route.ts POST (bulk tag) │ └── sync/route.ts POST (force full re-sync from GHL) ├── campaigns/ │ ├── route.ts GET (list), POST (create draft) │ ├── [id]/route.ts GET, PATCH, DELETE │ ├── [id]/send/route.ts POST (send now or schedule) │ ├── [id]/cancel/route.ts POST │ └── [id]/preview/route.ts POST (returns rendered HTML + audience count) ├── sequences/ │ ├── route.ts GET (list mirrored from GHL) │ └── [id]/enroll/route.ts POST (enroll a contact via tag trigger) ├── conversations/ │ ├── route.ts GET (threads list) │ ├── [id]/route.ts GET (messages in thread) │ ├── [id]/send/route.ts POST (send SMS reply) │ └── [id]/read/route.ts POST (mark read) ├── dashboard/ │ └── route.ts GET (aggregate widgets) └── settings/ └── route.ts GET, PATCH (sender identity, etc.) app/api/webhooks/ghl/ ├── contacts/route.ts Contact created / updated / deleted / tagged ├── events/route.ts Email + SMS events (opened, clicked, bounced, delivered) ├── conversations/route.ts Inbound SMS messages └── workflows/route.ts Sequence enrollment + completion events

Webhook security

  • Each webhook URL registered with GHL has a shared secret (env: GHL_WEBHOOK_SECRET).
  • Every incoming request is verified via HMAC-SHA256 signature in the X-GHL-Signature header.
  • Requests without a valid signature are rejected with 401 and logged.
  • Duplicate events (same ghl_event_id) return 200 immediately without re-processing (idempotent).

Key interaction flows

CSV Import (contacts/import)

1
User drops CSV. Browser parses first 5 rows for preview.
2
Column mapper auto-detects: Contact Name → name, Phone → phone, Email → email, Business Name → companyName, Tags → tags. User can override any mapping or mark a column as "ignore."
3
Validation pass: phone numbers normalised to E.164 (0412… → +61412…); emails lowercased; duplicate detection against existing marketing_contacts by email then phone.
4
Submission hits POST /api/marketing/contacts/import. Server streams rows to GHL via contacts.bulkUpsert in batches of 50. Per-batch result (created / updated / failed) returned as SSE to the UI for a live progress bar.
5
Each successful upsert mirrors into marketing_contacts. Failures collected into a downloadable error report CSV.

Send Campaign (campaigns/new)

1
Who — user picks: all contacts, one or more tags, or a smart segment. Live count shown by calling GHL contacts.search with the filter.
2
What — paste Claude-generated HTML into a Monaco-style editor. Preview pane renders both a desktop and mobile view via iframe sandbox. Subject + preheader inputs separately.
3
When — Send Now / Schedule / Recurring. Date picker in Australia/Brisbane. Recurring rules serialised as RRULE-ish JSON, translated to GHL format at submit time.
4
Review — final audience count, from identity preview, test-send button (sends to the logged-in user's email only), and a big Send button.
5
Send triggers POST /api/marketing/campaigns/[id]/send which calls GHL's bulk-action create endpoint, stores the returned bulk_action_id against the campaign row, and flips status to sending (or scheduled).

7. Contact Sync Logic

Initial sync (run once during M1)

  1. A one-off script scripts/ghl-initial-contact-sync.ts paginates through GHL contacts.search (100/page).
  2. Each contact is upserted into marketing_contacts with source = 'ghl_initial_sync'.
  3. Tag list preserved verbatim. Custom fields flattened into the custom_fields jsonb column.
  4. 338 contacts → ~4 pages → expected runtime < 10s.

Ongoing sync (steady state)

  • CC → GHL (writes): every admin mutation goes through a CC API route, which first calls GHL, then mirrors the result locally. GHL response is authoritative — if the GHL write fails, the local row is not touched.
  • GHL → CC (webhooks): the /api/webhooks/ghl/contacts handler upserts on contact.create / contact.update / contact.tag_added / contact.tag_removed / contact.dnd.
  • Reconciliation job: a manual "Force re-sync" button in settings calls /api/marketing/contacts/sync which runs the initial-sync script again. Upserts are idempotent so it's safe to run whenever.

Conflict rules

FieldWinnerRationale
Email, phone, name, company, tags, DND flagsGHLGHL is source of truth for marketing data.
profile_id (link to ChemConnect customer)CCGHL doesn't know about ChemConnect customers.
DeletionsGHLA webhook contact.delete soft-deletes in CC (sets deleted_at), preserving historical campaign stats.

8. Event Pipeline (Analytics)

GHL campaign fires -> email sent -> recipient opens | v GHL webhook POST -> /api/webhooks/ghl/events | v verify HMAC signature | v is ghl_event_id already in marketing_events? ---yes--> 200 OK (idempotent) | no v INSERT INTO marketing_events (event_type, campaign_id, contact_id, occurred_at, metadata) | v update marketing_campaigns aggregates (opened_count, etc.) via trigger | v update marketing_contacts.is_opted_out on unsubscribed/complained

Dashboard queries

  • Campaign card metrics — direct count queries against marketing_events filtered by campaign_id and event_type.
  • Monthly summary widgets — materialised view refreshed every 5 minutes; cheap to render.
  • Per-contact timeline — SELECT * FROM marketing_events WHERE contact_id = $1 ORDER BY occurred_at DESC LIMIT 50.
  • Deep analytics (best-time heatmap, per-link breakdown) — not calculated in CC. UI shows a "Open full analytics in GoHighLevel" button that deep-links to https://app.gohighlevel.com/v2/location/{locationId}/campaigns/{ghl_id}.

9. Two-Way SMS Inbox Logic

This is the most stateful part of the module — data is actively mutated from both sides.

Inbound flow

1
Recipient sends SMS to the CQVS number.
2
GHL receives via Twilio, fires conversation.message.inbound webhook to /api/webhooks/ghl/conversations.
3
CC looks up the contact by ghl_contact_id. If not found (unknown number), auto-creates a new marketing_contacts row so we never lose a lead.
4
Upsert sms_conversations row, insert sms_messages row with direction='inbound', bump unread_count, update last_message_at.
5
If body matches STOP / UNSUBSCRIBE / OPT OUT regex, set is_opted_out = true on the contact. GHL already does this on its side — we mirror so CC's suppression badges show correctly.
6
Supabase Realtime broadcasts the new row; open admin tabs update the inbox in real time with a red badge.

Outbound flow (reply from CC)

1
User types a reply in the conversation view, hits send.
2
POST /api/marketing/conversations/[id]/send optimistically inserts a local sms_messages row with status='queued'.
3
Backend calls GHL conversations.sendSms with the contact's phone and the body.
4
GHL returns a message_id; we store it against the local row and flip status to sent.
5
Later, GHL fires a sms.delivered or sms.failed webhook; the event lands in marketing_events and the local sms_messages.status is updated.
Fallback: If GHL's conversations-send endpoint is unexpectedly restricted even on the Unlimited plan, the outbound flow is shut off with a flag and the UI shows "Reply from GoHighLevel" deep-link instead. This was the risk flagged in the Final Integration Plan — first thing validated on day 1 of M3.

10. Sequences & Workflow Enrollment

Sequences are designed and live inside GHL. CC provides a thin enrollment layer only.

How a sequence fires from CC

  1. User tags a contact in CC as Lead, VIP, etc.
  2. CC POST /api/marketing/contacts/[id]/tags adds the tag via GHL contacts API.
  3. If the tag is linked to a workflow (mapping stored in admin_settings: sequences.tag_map), CC additionally calls GHL workflows.enrollContact explicitly — this is redundant with GHL's own tag-triggered workflows but guarantees enrollment even if GHL's trigger lags.
  4. GHL executes all steps (delays, branching, conditions) server-side. CC has zero execution responsibility.
  5. When a contact finishes a sequence, GHL fires a webhook; CC logs it to marketing_audit_log.

Lead Nurture seed sequence (configured in GHL during M4)

  • Consent source tracked — every contact row records how it got there (CSV import / existing customer / manual / GHL sync).
  • Unsubscribe link — GHL injects automatically; we verify by sending ourselves a test email on day 1 of M2.

  • Audit trail — every send, every tag change, every sequence enrollment logged to marketing_audit_log with actor + timestamp.
  • Data export — a contact's full record (profile + events + messages) exportable as JSON from their profile page, satisfying data-access requests.

  • Connection test — npm run ghl:test already in place. Will run in CI as a smoke check.
  • Webhook replay — a small scripts/replay-webhook.ts for re-posting captured GHL payloads locally against the dev server during development.

15. Risks & Mitigations

RiskMitigation
LC Phone activation delays (Stripe connection or support ticket) M3 is sequenced after M2 deliberately. If delay reaches day of M3 kickoff, we start M4 (Sequences) in parallel and circle back.
GHL API schema change during build All responses validated through Zod schemas at the boundary; schema failures surface clearly rather than corrupting downstream code.
Webhook delivery lag or loss Reconciliation job + manual "Force re-sync" button in settings. Idempotent handlers mean replays are safe.
Private Integration token revoked Token stored in ghl_oauth_tokens table, rotation documented. Settings page shows connection health; failed calls alert via the error-alert email.
Rate limits on 338-contact initial sync Negligible at this volume; even so, batches are 50 with 100ms spacing — well under documented GHL limits.
Outbound SMS reply endpoint restricted Validated on day 1 of M3. If restricted, feature-flag off the reply box and show deep-link to GHL instead. All other SMS features still work.
DNS propagation delays at go-live SPF/DKIM/DMARC likely already green on send.cqvs-chemconnect.com.au because GHL manages it. We verify via mxtoolbox on day of go-live; any issues flagged 24h early.
