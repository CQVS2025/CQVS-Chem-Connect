1. MacShip Integration (Live Shipping)

1.1 Customer-Facing Changes

Product Pages

  • Remove "Shipping: $X / Free" display
  • Remove "Available in [Region]" section
  • Keep "Ships from your state" delivery info text

Cart Page

  • Remove per-product shipping fee breakdown
  • Show: "Shipping calculated at checkout"

Checkout Page

  • Customer enters delivery address including postcode
  • New required dropdown: "Forklift available on site" / "No forklift on site"
  • Once postcode is entered, system calls MacShip API for a live shipping quote
  • Shipping cost + container cost appear in the order summary
  • If postcode not serviceable: show "We can't calculate shipping for your area" with a submit button that emails admin

1.2 How Shipping Works (4-Step Flow)

1
Customer Enters Postcode - System calls MacShip Get Routes API. Sends: pickup warehouse (closest to customer), delivery postcode, item details (packaging size, weight, dimensions), DG classification, tailgate requirement (YES if no forklift). Returns carrier options with prices. We always pick the cheapest carrier.
POST /apiv2/routes/returnRoutes
2
Customer Places Order - System calls MacShip Create Consignment API to book the shipment. Gets back: MacShip consignment ID, carrier ID, tracking info. All stored on the order.
POST /apiv2/consignments/createConsignment
3
Admin Dispatches - When admin updates order status, system calls MacShip Manifest API. Carrier is notified to pick up goods. Labels auto-generated and printable from admin dashboard.
POST /apiv2/manifests/manifest
4
Live Tracking - System periodically syncs tracking updates and POD (Proof of Delivery) from MacShip. Visible to both customer and admin.
GET /apiv2/consignments/getConsignment?id={id}

1.3 Warehouse Selection Logic

  1. Check which warehouses carry the ordered product (based on product's region setting)
  2. Pick the closest warehouse to the customer's postcode
  3. Use that warehouse as the pickup location
  4. If order has products from different warehouses: ship everything from one warehouse (the closest that has all products), even if further away

1.5 DG (Dangerous Goods) Handling

Each product already has a DG classification (Non-DG, DG Class 3, 5, 6, 8). This is included in every MacShip quote request so carrier pricing accounts for DG transport and documentation requirements. DG products cost more to ship.

2. Xero Integration (Accounting)

2.1 Automatic Invoicing (Purchase Order Customers)

Current Flow (Manual)

  1. Customer pays by Purchase Order on our platform
  2. You manually create an invoice in Xero, add PO number, send to customer

New Flow (Automated)

  1. Customer pays by Purchase Order on our platform
  2. System checks if customer exists as a Xero Contact (by email)
  3. If not, creates them as a new Xero Contact automatically
  4. Creates an Invoice in Xero with:
    • Customer's PO number as reference
    • All line items (product name, qty, unit price, packaging size)
    • Container costs (IBC/drum) as separate line items
    • Shipping cost (from MacShip)
    • GST (10%)
    • Due date: 30 days net
  5. NEW The uploaded PO document is attached to the Xero invoice
  6. NEW Invoice is sent to the customer's invoice email address (may differ from their login email - e.g., accounts team)
  7. Invoice is auto-sent (no manual review needed)
PUT https://api.xero.com/api.xro/2.0/Invoices (Type: ACCREC, Status: AUTHORISED)

2.2 Automatic Purchase Orders (to Warehouses)

New Flow (Automated)

  1. Customer places order on our platform
  2. System determines the closest warehouse (based on product region + postcode)
  3. Creates a Purchase Order in Xero to that warehouse with:
    • Warehouse as the Xero Contact (supplier)
    • Product details (name, qty, packaging size)
    • Warehouse-specific pricing (configured in admin)
    • Container costs (warehouse-specific)
    • Customer's delivery address
    • Reference to our platform order number
  4. Purchase Order is sent to the warehouse via Xero
PUT https://api.xero.com/api.xro/2.0/PurchaseOrders

2.3 Customer Contact Sync

When a new customer signs up, we automatically create them as a Xero Contact using the registration data (company name, contact name, email, phone, ABN, address). No extra data needed - we already collect everything required.

PUT https://api.xero.com/api.xro/2.0/Contacts

2.4 Xero Authentication

OAuth 2.0 flow. "Connect to Xero" button in admin dashboard. One-click authorization. We handle token refresh automatically so the connection stays active.

3. Pricing & Packaging Size Rework

Major Change: The pricing model is being updated from a single price per product to pricing per packaging size, with per-litre calculation and warehouse-specific costs.

3.1 Packaging Size Pricing NEW

  • Each product can have multiple packaging sizes (20L Drum, 200L Drum, 1000L IBC, etc.)
  • Each packaging size has its own price per litre
  • Example: Green Acid Replacement
    • 20L Drum: $2.80/L - total = $56.00
    • 200L Drum: $2.45/L - total = $490.00
    • 1000L IBC: $2.10/L - total = $2,100.00
  • Packaging sizes are managed as a dropdown selector (not free text) in admin
  • Admin can configure which packaging sizes are available per product

3.2 Price Calculation CHANGE

ScenarioCalculationExample
Per-litre productPrice/L x Packaging Size Litres x Qty$2.45/L x 20L x 1 = $49.00
Fixed-price productFixed Price x Qty$200.00 x 1 = $200.00

Each product will have a price type flag: "Per Litre" or "Fixed Price". Admin sets this when creating/editing products.

3.3 Container Costs NEW

  • When customer orders a product in 1000L IBC or 200L Drum, there is a container cost (the physical container itself)
  • Container cost is a separate line item added automatically at checkout (after postcode is entered and warehouse is determined)
  • Container costs vary by warehouse (e.g., VIC warehouse: IBC = $200, QLD warehouse: IBC = $250)
  • Admin configures container costs per warehouse per packaging size
  • Customer sees: "IBC Container - $200.00" as a separate line in their order summary

3.4 Warehouse-Specific Pricing NEW

  • Each warehouse may charge different prices for the same product
  • Admin configures this in the warehouse management section
  • These prices are used for Xero Purchase Orders (what you pay the warehouse)
  • Customer always sees the customer-facing price (what they pay you)
