1. Overall Assessment

Treating the supplier as a "special kind of warehouse" is the right call — and even cleaner than the doc suggests, because the existing warehouses table already carries everything we need. contact_name, contact_email, contact_phone, xero_contact_id, address fields, is_active — all there. product_warehouses already handles the multi-product-per-warehouse mapping, and warehouse_product_pricing already gives us the cost-side data we need for Xero reconciliation.

What's new is essentially: a flag on warehouses, a freight strategy switch in lib/macship/warehouse-selector.ts (which becomes a more general router), a rate-sheet model, a few new columns on orders for supplier dispatch state, and the supplier dashboard. The bones already exist.

Three things worth calling out that you got right:

  • Buyer pays Chem Connect, we reconcile in Xero. No marketplace splits, no separate Stripe accounts. Sticks with the existing pattern.
  • Supplier dashboard is read-mostly. Limiting what the supplier can do (no refunds, no product edits, no other customers visible) is exactly right.
  • The new path runs alongside MacShip, not over it. Zero impact on existing products and existing orders. Non-negotiable.

3. Answers to the Nine Questions

Answer 1 Which freight rate column should we use for AdBlue?

Bulk (Post 14Jul25) for the tanker pump-in flow. We're well past the July 2025 cutover, so post-July rates apply. The Pack columns are not in scope for Phase 1.

Important caveat: the freight engine must support multiple rate sheets and multiple unit types from day one, even if only Bulk goes live in Phase 1. See Section 5 — the matrix has three product variants, not one, and two different calculation models. Wiring this in later is a rebuild, not an extension.

Action: The freight engine reads a rate sheet keyed off product + packaging size (not just product). Phase 1 only wires the Bulk rate sheet. Pack Existing and Pack New rate sheets get added later as configuration, not code.

Answer 2 Freight origin for distance calculations?

Option (c) — supplier confirms depot manually after the order arrives, with a default origin used to pre-show an estimate at checkout.

The default origin postcode lives on the supplier's warehouses row, in the existing address_postcode field. The estimate at checkout uses that postcode; the supplier can override the dispatch depot in their dashboard once the order lands. Locking in option (b) would force us to maintain depot-level stock state inside Chem Connect, which we're not equipped to do.

Action: Default origin = warehouses.address_postcode for the supplier-managed warehouse row. Final freight is reconfirmed by the supplier on dispatch. If the final number differs from the indicative quote by more than a set threshold, flag it and notify the buyer before dispatch.

Answer 3 What if the buyer answers "No" to truck access?

A blend of (a) and (c). Show a warning at checkout, but still let the order through. Surface the answer prominently on the supplier dashboard so the supplier calls the buyer before dispatch.

Suggested checkout copy: "Our standard delivery is a 19-metre tanker. If this truck cannot access your site, the supplier will contact you to arrange an alternative, which may involve additional cost or delay."

Hard-blocking the order loses sales and frustrates buyers who almost certainly have a workaround (smaller truck, different gate, scheduled access). That's a supplier conversation, not a checkout-killer.

Future enhancement: if/when we add packed AdBlue (IBC), a "No" to truck access could nudge the buyer toward IBC delivery instead. Conversion-saver. Not Phase 1, but worth modelling so we don't paint into a corner.

Answer 4 What other site-access questions?

For tanker pump-in, the question set should be:

  • Can a 19-metre rigid/tanker truck access the delivery site? (already proposed)
  • Tank capacity and approximate empty space at time of delivery (so the supplier doesn't roll up with 5,000L for a tank that fits 3,000L)
  • Fill point location and approximate hose run distance from where the truck can park
  • Site operating hours and after-hours contact number
  • Site induction or sign-in requirement (yes/no, plus a free-text notes field)
  • Hardstand suitable for a loaded tanker (yes/no — some yards are soft)
  • Existing tank fitting type (camlock size, dry-break, other — supplier to confirm what fittings their hoses carry, then we offer those as options)

Forklift: drop from Phase 1. It's only relevant if we sell IBC AdBlue, which isn't in Phase 1. Re-introduce alongside the IBC variants.

Action: Site-access questions need to be configured per product or per product+packaging size, not hardcoded into checkout. Suggested approach: a product_checkout_questions table with product_id, packaging_size_id (nullable for "all sizes"), question text, type (yes/no, text, select), required (boolean). Answers stored on the order as site_access_answers jsonb. Same engine handles future supplier-specific questions.

Concrete plants run on tight chemical inventory. Knowing when product leaves and when it arrives lets them plan their day, decide whether to be on-site to receive it, and gives them a clearer commitment from the supplier. It's also a tighter feedback loop for me operationally — if dispatch slips, I want to know before delivery slips.

5. Findings From the Freight Matrix — Important

I went through the freight matrix carefully. The sheet is more structured than the doc assumes, and a few things need to be designed in now to avoid rework later.

5.1 What the matrix actually contains

ColumnUnitUsed for
Bulk (Pre-13Jul25)$ per litreBulk tanker pump-in, old rate (legacy reference only)
Bulk (Post-14Jul25)$ per litrePhase 1 column. Bulk tanker pump-in, current rate
Pack (Pre-13Jul25)$ per consignmentPacked AdBlue (IBC), refill of customer's existing IBC, old rate
Pack Existing (Post-14Jul25)$ per consignmentPacked AdBlue (IBC), refill of existing IBC, current rate
Pack New (Pre-13Jul25)$ per consignmentPacked AdBlue (IBC), new IBC supplied with product, old rate

Distance is bracketed in 100km steps from 0–100 up to 3901–4000. There is also a 4001–4100 row with no rate data, which we need to clarify with the supplier (see 5.6).

Pack Existing vs Pack New: the price gap is consistently ~10% (e.g. $63 vs $70 at 0–100km). This is almost certainly the standard liquid-chemicals pattern of "refill existing IBC = cheaper" vs "new IBC supplied = more expensive because container is included". Worth confirming with the supplier, but it's the only structure that fits.

5.2 The freight engine has to handle two unit types

This is the key one. Your original doc treats freight as "look up a single column for AdBlue". The matrix has two fundamentally different calculation models:

  • Bulk: rate per litre × order litres × distance bracket lookup
  • Pack: flat rate per consignment × distance bracket lookup (does not multiply by litres)

Each rate sheet needs to declare its calculation model. Each product+packaging combination needs to point to a rate sheet. The freight engine reads both.

5.3 Three product variants, not one

Even within AdBlue alone, we have:

  1. Bulk pump-in (litres into a tank)
  2. Packed AdBlue, refill of customer's existing IBC
  3. Packed AdBlue, new IBC supplied

Looking at the seed data in schema.sql, the existing AdBlue product already has packaging_sizes set to {'10L Jerry Can', '200L Drum', '1000L IBC'}. Realistically only the IBC sizes will use the pack rate sheets, and they need to know whether they're a refill (Pack Existing) or new container (Pack New).

Phase 1 only goes live for variant 1. But variants 2 and 3 are very likely — not every site has a bulk tank, and IBC sales are common. If the architecture treats AdBlue as a single product with a single rate sheet, we end up rebuilding to add variants 2 and 3.

5.4 Suggested data model (sketch, not prescriptive)

To make this concrete, here's roughly the shape I'd expect. Adjust as you see fit:

-- Extend warehouses
alter table warehouses
  add column if not exists is_supplier_managed boolean not null default false;

-- A rate sheet describes a calculation model (per-litre, flat per consignment,
-- per-pallet, per-kg, etc.) plus its origin and active status.
create table supplier_rate_sheets (
  id uuid primary key default uuid_generate_v4(),
  warehouse_id uuid not null references warehouses on delete cascade,
  name text not null,                           -- 'AdBlue Bulk Post-14Jul25'
  unit_type text not null,                      -- 'per_litre' | 'flat_per_consignment' | 'per_kg' | 'per_pallet'
  origin_postcode text,                         -- defaults to warehouses.address_postcode if null
  is_active boolean not null default true,
  created_at timestamptz not null default now()
);

-- Distance brackets and rates for each rate sheet
create table supplier_rate_sheet_brackets (
  id uuid primary key default uuid_generate_v4(),
  rate_sheet_id uuid not null references supplier_rate_sheets on delete cascade,
  distance_from_km integer not null,
  distance_to_km integer not null,
  rate numeric(10, 4) not null
);

-- Map a product + packaging size combo to a rate sheet.
-- packaging_size_id NULL means "applies to all sizes for this product".
create table product_freight_rate_sheets (
  product_id uuid not null references products on delete cascade,
  packaging_size_id uuid references packaging_sizes on delete cascade,
  rate_sheet_id uuid not null references supplier_rate_sheets on delete cascade,
  primary key (product_id, packaging_size_id)
);

-- New columns on orders for the supplier-managed flow
alter table orders
  add column if not exists supplier_dispatch_date date,
  add column if not exists supplier_dispatch_notes text,
  add column if not exists supplier_tracking_url text,
  add column if not exists supplier_freight_cost numeric(10, 2),
  add column if not exists site_access_answers jsonb;

5.5 Where this plugs into existing code

The cleanest place for the freight strategy split is lib/macship/warehouse-selector.ts, which becomes a more general fulfillment router. Roughly:

  • If the selected warehouse has is_supplier_managed = false ⇒ current MacShip path runs unchanged.
  • If is_supplier_managed = true ⇒ new SupplierManagedStrategy is invoked, which looks up the rate sheet for each cart line, computes the freight cost, and skips the MacShip quote call entirely.

lib/orders/calculate.ts already calls a warehouse selector and handles the freight number cleanly. The supplier-managed path returns a freight number the same way MacShip does — the rest of the order math (GST, processing fee, total) works without changes.

5.6 Open items to confirm with the supplier

  • Pack Existing vs Pack New meaning. Confirm it means refill of customer-owned IBC vs new IBC supplied with product. (My interpretation, not yet validated with the supplier.)
  • 4001+ km row is empty. Either the supplier doesn't service past 4000km, or the data is missing. Decide what the checkout does for buyers past that distance — block, "quote on application", or fall back to last bracket?
  • Minimum freight charge. A small bulk drop close to the depot is still cents-per-litre at low volume. Is there a freight floor or minimum order size?
  • Distance calculation method. Confirm we use a road-distance API (Google Distance Matrix or similar), not straight-line. And confirm the bucket logic rounds up to the next 100km bracket (so 350km uses the 301–400 row).

5.7 Knock-on for future suppliers (Question 9)

If even AdBlue alone has two calculation models (per-litre and flat), and other suppliers may use per-pallet, weight-based or zone-based pricing, the freight engine cannot be a single-column lookup. The unit_type field on rate sheets is the lever — it needs to support at least:

  • per_litre × distance (Bulk AdBlue)
  • flat_per_consignment × distance (Pack AdBlue)
  • per_kg or per_pallet × distance, or × zone (likely future suppliers)

This is a Phase 1 architectural decision, not a Phase 2 one. Building it in now is far cheaper than retrofitting.
